## 1. Purpose

franky-golang provides a purpose-built subagent for Go development
work within franky's agent orchestration framework.

Two overlapping use cases:

### 1.1 — Custom preset via `subagent` tool

A subagent preset named `"go-dev"` bundles 6 file/shell tools from
the parent (`read`, `write`, `edit`, `ls`, `bash`, `grep`) plus 3
Go-specific convenience tools (`gofmt`, `govet`, `gotest`). The LLM
chooses this preset via `subagent({"preset": "go-dev", ...})` to
delegate Go development tasks to a focused subagent with a
Go-oriented system prompt.

### 1.2 — Parent LLM custom tools

The 3 Go tools are also registered as top-level tools available to
the parent LLM when the extension is loaded. The parent can call
`gofmt`, `govet`, or `gotest` directly without spawning a subagent.

### 1.3 — Reference pattern

Beyond its own utility, franky-golang is a **reference implementation**
for any franky extension that wants to register custom subagent
presets. Every extension that ships custom tools + a custom preset
follows the same shape:

```
extensions_builtin/$name.zig
  └── pub fn extension() ext.Extension
        └── fn init(ext, host)
              ├── host.registerTool(myTool1())
              ├── host.registerTool(myTool2())
              └── host.registerPreset(.{
                    .name = "my-preset",
                    .build_tools = buildMyTools,
                    ...
                  })
```

## 2. Package topology

```
franky-golang/
├── build.zig                       — Module definition + tests + binary
├── build.zig.zon                   — Depends on franky via `{ .path = "../" }`
└── src/
    ├── go_dev.zig                  — The extension module
    │     ├── gofmtTool()           — AgentTool factory
    │     ├── govetTool()           — AgentTool factory
    │     ├── gotestTool()          — AgentTool factory
    │     ├── registerPreset(reg)   — Standalone registration helper
    │     ├── extension()           — Ext.Extension factory (Tier-1)
    │     ├── buildGoDevTools()     — Preset builder (internal)
    │     └── selectTools()         — Tool filter (internal)
    └── main.zig                    — Optional standalone binary
```

### 2.1 Dependencies

- **franky** (`{ .path = "../" }` in build.zig.zon) — the host SDK
  providing `agent.types.AgentTool`, `ai.types.ContentBlock`,
  `ai.stream.Cancel`, `coding.tools.subagent.*`,
  `coding.extensions.*`, `coding.slash.*`.

### 2.2 Exports

`go_dev.zig` exposes 4 public symbols:

| Symbol | Type | Purpose |
|--------|------|---------|
| `gofmtTool()` | `fn () at.AgentTool` | Factory for the `gofmt` tool stub |
| `govetTool()` | `fn () at.AgentTool` | Factory for the `govet` tool stub |
| `gotestTool()` | `fn () at.AgentTool` | Factory for the `gotest` tool stub |
| `registerPreset(reg)` | `fn (*PresetRegistry) !void` | One-shot registration of the `"go-dev"` preset into any registry |
| `extension()` | `fn () ext.Extension` | Tier-1 extension factory for the built-in catalog system |

## 3. Types & surface

### 3.1 `AgentTool` (from franky core)

Each tool is an `agent.types.AgentTool` struct:

```zig
pub const AgentTool = struct {
    name: []const u8,
    description: []const u8,
    parameters_json: []const u8,
    execution_mode: ExecutionMode = .parallel,
    ctx: ?*anyopaque = null,
    execute: *const fn (
        tool: *const AgentTool,
        allocator: std.mem.Allocator,
        io: std.Io,
        call_id: []const u8,
        args_json: []const u8,
        cancel: *ai.stream.Cancel,
        on_update: at.OnUpdate,
    ) anyerror!ToolResult,
};
```

### 3.2 `Extension` (from franky core)

```zig
pub const Extension = struct {
    name: []const u8,
    version: []const u8 = "0.0.0",
    userdata: ?*anyopaque = null,
    init_fn: ?*const fn (ext: *Extension, host: *Host) ExtError!void = null,
    deinit_fn: ?*const fn (ext: *Extension) void = null,
};
```

### 3.3 `Host` (from franky core, extended with preset support)

```zig
pub const Host = struct {
    allocator: std.mem.Allocator,
    slash_registry: *slash.Registry,
    tools: *std.ArrayList(at.AgentTool),
    subscriptions: *std.ArrayList(Subscription),
    presets: ?*subagent_mod.PresetRegistry,   // v2.5 — SDK hook

    pub fn registerCommand(...) !void;
    pub fn registerTool(...) !void;
    pub fn subscribe(...) !void;
    pub fn registerPreset(p: subagent_mod.Preset) !void;  // v2.5 — SDK hook
};
```

### 3.4 `Preset` (from franky core)

```zig
pub const Preset = struct {
    name: []const u8,
    description: []const u8,
    default_profile: []const u8,
    default_role: role_mod.Role,
    default_system_prompt: []const u8,
    build_tools: *const fn (allocator, parent_tools) ![]at.AgentTool,
    safety: SafetyClaims = .{},
};
```

## 4. Tool schemas

### 4.1 `gofmt`

| Field | Value |
|-------|-------|
| name | `"gofmt"` |
| description | `"Run gofmt -d on a Go file to show formatting diffs. Pass the file path as \`path\`."` |
| parameters_json | `{"type":"object","required":["path"],"properties":{"path":{"type":"string","description":"Path to the Go file to format-check"}}}` |
| execution_mode | `.sequential` |
| execute | `gofmtExecute` — shells out to `gofmt -d <path>`, returns diff |

**Planned real impl:** Shell out to `gofmt -d <path>`, capture stdout
(the diff), return it as tool result text. Error on non-Go files,
non-existent paths, or `gofmt` not in `$PATH`.

### 4.2 `govet`

| Field | Value |
|-------|-------|
| name | `"govet"` |
| description | `"Run go vet on a Go package to check for suspicious constructs. Pass the package path as \`pkg\`."` |
| parameters_json | `{"type":"object","required":["pkg"],"properties":{"pkg":{"type":"string","description":"Package path (e.g. ./... or ./internal/foo)"}}}` |
| execution_mode | `.sequential` |
| execute | `govetExecute` — shells out to `go vet <pkg>`, returns output |

**Planned real impl:** Shell out to `go vet <pkg>`, capture stdout +
stderr. Return vet findings as text. Error on non-existent paths,
or `go` not in `$PATH`.

### 4.3 `gotest`

| Field | Value |
|-------|-------|
| name | `"gotest"` |
| description | `"Run go test on a Go package and report results. Pass \`pkg\` (required) and optionally \`flags\`."` |
| parameters_json | `{"type":"object","required":["pkg"],"properties":{"pkg":{"type":"string","description":"Package path (e.g. ./...)"},"flags":{"type":"string","description":"Additional go test flags (optional, e.g. -v -race)"}}}` |
| execution_mode | `.sequential` |
| execute | `gotestExecute` — shells out to `go test <flags> <pkg>`, returns output |

**Planned real impl:** Shell out to `go test <flags> <pkg>`, capture
stdout + stderr + exit code. Return test output. Support `-v` for
verbose, `-race` for race detector, etc. via the `flags` arg.

## 5. Preset definition

The `"go-dev"` preset:

```zig
{
    .name = "go-dev",
    .description = "Go development: edit, format, vet, and test Go code.",
    .default_profile = "",                // inherit parent's profile
    .default_role = .code,               // can run bash
    .default_system_prompt =
        \\You are a Go development sub-agent. Your job is to help develop,
        \\review, and maintain Go code.
        \\...
        \\Workflow:
        \\  1. Read the relevant files first.
        \\  2. Make focused edits.
        \\  3. Run gofmt on every Go file you create or modify.
        \\  4. Run go vet on affected packages.
        \\  5. Run go test to verify correctness.
        \\  6. Report what you did and what the tool output says.
        ,
    .build_tools = buildGoDevTools,
}
```

### 5.1 Tool selection (`buildGoDevTools`)

The builder selects 6 tools from `parent_tools` by name:

| Tool | Source | Purpose |
|------|--------|---------|
| `read` | parent_tools | Read files |
| `write` | parent_tools | Create files |
| `edit` | parent_tools | Edit existing files |
| `ls` | parent_tools | List directories |
| `bash` | parent_tools | Run shell commands |
| `grep` | parent_tools | Search file contents |

Then appends 3 custom tools (freshly constructed, not from parent):

| Tool | Source | Purpose |
|------|--------|---------|
| `gofmt` | `gofmtTool()` | Format Go code |
| `govet` | `govetTool()` | Vet Go code |
| `gotest` | `gotestTool()` | Test Go code |

Total: **9 tools**.

### 5.2 Role semantics

- `default_role = .code` — the subagent can execute `bash` (for
  `go build`, `go test`, etc.) and use all file tools.
- Parent must be at least `.code` role to spawn a `"go-dev"` subagent.
- If parent role is lower (`.plan` or `.read`), `runSubagent` returns
  `role_promotion_denied` with a hint.

### 5.3 Profile inheritance

- `default_profile = ""` — instructs the subagent dispatch to use
  the parent's profile (the `--profile` value the user started with).
- This means the Go subagent uses whatever provider the user chose,
  without hardcoding a provider name in the preset.

## 6. Wire path at runtime

### 6.1 Registration path (Tier-1 extension)

```
1. User runs:  franky --extensions go-dev "prompt"

2. Mode driver (interactive.zig):
   a. Init PresetRegistry
   b. registerBuiltinPresets(&reg)
   c. Init Extension Manager:  mgr.presets = &reg
   d. Parse --extensions list → ["go-dev"]
   e. Lookup "go-dev" in catalog → go_dev.extension()
   f. mgr.register(ext, &slash_registry)
        → init_fn(&ext_, &host)
            ├── host.registerTool(gofmtTool())   → mgr.host_tools += gofmt
            ├── host.registerTool(govetTool())   → mgr.host_tools += govet
            ├── host.registerTool(gotestTool())  → mgr.host_tools += gotest
            └── host.registerPreset(go-dev)      → reg.register("go-dev", ...)
   g. mgr tools are merged into binding.tools
   h. buildParametersJson(&reg) now includes "go-dev" in enum
   i. Wire subagent Ctx with .presets = &reg
   j. Wire list_subagent_presets tool (LLM discovery)

3. LLM sees "go-dev" in list_subagent_presets.
4. LLM calls subagent({"preset": "go-dev", "prompt": "..."}).
5. runSubagent resolves preset, calls buildGoDevTools, spawns 9-tool subagent.
```

### 6.2 Registration path (standalone — no extension system)

For modes that don't load extensions (print, rpc, proxy):

```zig
var reg = subagent_mod.PresetRegistry.init(allocator);
defer reg.deinit();
try subagent_mod.registerBuiltinPresets(&reg);
try go_dev.registerPreset(&reg);            // ← one-shot
const params = try subagent_mod.buildParametersJson(allocator, &reg);
// ... proceed with subagent Ctx as usual ...
```

### 6.3 Discovery

The LLM discovers the `"go-dev"` preset via the `list_subagent_presets`
tool:

```json
// Response from list_subagent_presets:
[
  {"name":"research","description":"...","default_profile":"gemini","default_role":"read","read_only":true},
  {"name":"code-audit","description":"...","default_profile":"gemini","default_role":"read","read_only":true},
  {"name":"file-ops","description":"...","default_profile":"","default_role":"plan","read_only":false},
  {"name":"bash-runner","description":"...","default_profile":"","default_role":"code","read_only":false},
  {"name":"go-dev","description":"Go development: edit, format, vet, and test Go code.","default_profile":"","default_role":"code","read_only":false}
]
```

### 6.4 Invocation

```json
// LLM → subagent tool:
{
  "preset": "go-dev",
  "prompt": "Add a health check handler to main.go. Format with gofmt, vet, and test."
}
```

### 6.5 Dispatch

1. `runSubagent` parses args, extracts `preset: "go-dev"`.
2. `ctx.presets.get("go-dev")` returns the custom preset.
3. `preset.build_tools(allocator, parent_tools)`:
   - `selectTools` filters: `[read, write, edit, ls, bash, grep]`
   - Appends: `[gofmt, govet, gotest]`
   - Returns 9-element `[]AgentTool`
4. `applyProfile` with `default_profile = ""` → inherits parent's profile.
5. Role check: `.code` requires parent ≥ `.code`.
6. Subagent spawned with Go system prompt, `.code` role, 9 tools.

## 7. SDK hook

### 7.1 What was added to franky core

The `src/coding/extensions.zig` module was extended with preset
registration support:

**`Host` struct** (new field + method):
```zig
pub const Host = struct {
    // ... existing fields ...
    presets: ?*subagent_mod.PresetRegistry,

    pub fn registerPreset(self: *Host, p: subagent_mod.Preset) !void {
        if (self.presets) |reg| try reg.register(p);
    }
};
```

**`Manager` struct** (new field):
```zig
pub const Manager = struct {
    // ... existing fields ...
    presets: ?*subagent_mod.PresetRegistry = null,
    // ...
};
```

The `Host` constructor in `Manager.register()` passes `self.presets`
through the field. When null, `registerPreset()` is a no-op.

### 7.2 Mode driver changes

Only `interactive.zig` currently loads extensions. The other three
modes (print, rpc, proxy) do not. To fully enable extension-wired
presets in a mode:

```zig
var ext_manager = extensions_mod.Manager.init(allocator);
ext_manager.presets = &preset_registry;    // ← SDK hook
defer ext_manager.deinit();
// ... load extensions as usual ...
```

**Why not wired in all 4 modes yet.** `interactive.zig` creates its
`PresetRegistry` inside `SessionBinding.init()`, which runs after
the extension manager. The scope disconnect is a known follow-up.

### 7.3 Null safety

Extensions that call `host.registerPreset()` when `Manager.presets`
is null get a **silent no-op** (the `?.` unwrap skips the call).
This means an extension loaded in a mode that hasn't wired the hook
will compile and run, but its presets won't appear. A future
improvement should `@panic` or return `ExtError.NotFound` when
`presets` is null and `registerPreset` is called.
